# Пояснение по техническим решениям
в тестовом задании на вакансию PHP-разработчика

## Используемый стек

- Laravel 12
- PHP 8.4
- SQLite
- Docker
- Laravel Breeze (Blade)
- Chart.js
- jenssegers/agent
- Vite

---

## Общие технические решения

### Docker

Проект запускается через Docker.

Причины:
- одинаковое окружение для разработчика и проверяющего;
- отсутствие проблем совместимости PHP/extensions;
- запуск проекта одной командой;
- отсутствие необходимости устанавливать PHP/Composer/Node.js локально.

---

### SQLite вместо MySQL

Для тестового задания SQLite проще и быстрее:
- не нужен отдельный DB-container;
- минимальная конфигурация;
- достаточно для небольшого объема данных.

---

### Особенности seeders

Seeders не сделаны полностью идемпотентными. Для `visits` не критично. Для тестового достаточно.

---

### Repository pattern

Для работы с БД используются Repository classes.

Причины:
- разделение бизнес-логики и работы с БД;
- упрощение поддержки;
- более чистая архитектура;
- соответствие принципу Single Responsibility.

Возможно, для тестового задания это избыточно, но показывает архитектурный подход.

---

## 1. Консольная команда + JSON API

### Внешний API

Используется:

```text
https://official-joke-api.appspot.com/random_joke
```

Причины:
- публичный API;
- не требует API key;
- простой JSON response.

---

### Scheduler

Используется Laravel Scheduler.

Команда:
```bash
php artisan joke:fetch
```

запускается каждые 5 минут через:

```php
Schedule::command('joke:fetch')->everyFiveMinutes();
```

---

### Почему используется schedule:work

Scheduler запускается внутри Docker container.

Используется:
```bash
php artisan schedule:work
```

Причины:
- проще для Docker;
- не требуется отдельный cron внутри контейнера;
- достаточно для тестового задания.

---

### Почему не используется Queue

Для тестового это избыточно, чуть усложняется логика и чуть сложнее для инфраструктуры.

Где бы использовали:
- для `joke:fetch`;
- для visits: `POST /api/visits` мог бы быстро принять запрос, а GeoIP/User-Agent обработать в фоне.

---

## 2. JS-фильтр полей формы

### Vanilla JavaScript вместо jQuery

Используется vanilla JS.

Причины:
- задача не требует тяжелых frontend libraries;
- меньше зависимостей.

---

### Особенности алгоритма отображения полей

1. Кнопки не учитываются, только доступные для заполнения поля
2. Атрибут `name` разбивается на части для исключения ошибок типа `1 совпадает с 12`
3. Разделение `name` сделано с универсальным разделителем для исключения ошибок типа `- вместо _` 
4. Дополнительно скрываются spacer elements (можно было более универсальный алгоритм, но для тестового это избыточно)

---

## 3. Дополнительное задание — статистика посещений

### Почему IP определяется на backend

IP определяется через:
```php
$request->ip()
```

Причины:
- frontend не может надежно определить IP;
- backend получает реальный client IP;
- стандартный подход аналитических систем.

---

### Почему используется GeoIP API

Для определения города используется внешний GeoIP API.

Причины:
- простота реализации;
- отсутствие необходимости локальной GeoIP database;
- достаточно для тестового задания.

При работе через hosting/proxy GeoIP может определять город по IP прокси-сервера. 
Для production потребовалась бы более точная настройка trusted proxies / headers.

---

### jenssegers/agent для User-Agent parsing

Используется:
```text
jenssegers/agent
```

Причины выбора:
- простой Laravel-friendly API;
- не самая высокая точность, но для тестового достаточно;
- минимальный объем кода.

---

### Почему используется visitor_hash

Для подсчета уникальных посещений используется:
```text
visitor_hash
```

На основе:
- IP
- User-Agent
- текущей даты

Причины:
- простая реализация unique visits;
- достаточно для тестового задания.

Для полноценного production analytics потребуется более сложная реализация.

---

### Почему Chart.js

Используется Chart.js через CDN.

Причины:
- популярная библиотека;
- простое подключение;
- достаточно для графиков тестового задания;
- не требует сложной frontend-инфраструктуры.

---

### Breeze (Blade) для авторизации

Используется Laravel Breeze.

Причины:
- минималистичная встроенная auth system;
- не требует лишних зависимостей;
- быстрое развертывание;
- достаточно для страницы статистики.
